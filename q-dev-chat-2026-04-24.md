## Core Modules

### 1. `src/server/demoAuth.ts` — Auth Layer
- No NextAuth/OIDC in demo mode. Uses a **custom HMAC-SHA256 signed cookie** (`br_demo_session`)
- `signDemoSession(session)` → base64url payload + HMAC signature → cookie value
- `verifyDemoSession(token)` → validates signature, returns `{username, role}`
- `getDemoSession()` → reads cookie from Next.js `cookies()`, verifies, returns session
- Secret comes from `AUTH_SECRET` env var (falls back to `"dev-secret-change-me"`)

### 2. `src/server/roles.ts` — RBAC
- Defines 5 roles with a numeric rank: `ENGINEER(10) < BENCH_MANAGER(40) < TALENT(50) < PRACTICE_LEAD(60) < COMPLIANCE(70)`
- `isUserRole()` — type guard
- `pickHighestRole()` — used when a user has multiple IdP roles

### 3. `src/middleware.ts` — Route Guard
- Runs on every request (except `/_next`, `/favicon`, `/api/demo/*`)
- Reads `br_demo_session` cookie, verifies it, checks role against `ROUTE_ALLOWLIST`
- Redirects to `/` (unauthenticated) or `/unauthorized` (wrong role)

```
/engineer          → ENGINEER only
/interview/:id     → ENGINEER, BENCH_MANAGER, PRACTICE_LEAD
/observer/*        → BENCH_MANAGER, PRACTICE_LEAD
/admin/*           → BENCH_MANAGER, PRACTICE_LEAD, TALENT
/talent            → TALENT only
/practice          → PRACTICE_LEAD only
/compliance        → COMPLIANCE only
```

### 4. `src/server/db.ts` — Prisma Singleton
- Standard Next.js Prisma singleton pattern — stores instance on `globalThis` to survive hot reloads in dev

### 5. `src/server/aiInterviewAssessment.ts` — Post-Interview Scoring
- Called when engineer clicks "Mark complete"
- With `OPENAI_API_KEY`: sends JD + resume summary + full transcript to OpenAI, gets back structured JSON scores
- Without key: runs `heuristicAssessment()` — scores by transcript length and turn count
- Output: `{technicalKnowledge: {score, rationale}, communication: {score, rationale}, proposedVerdict, summary, strengths[], gaps[]}`
- Merges result into `transcriptJson.meta.aiAssessment` and saves to DB

### 6. `src/app/api/interviews/[id]/next-question/route.ts` — Live Question Engine
- Called by the voice client after each answer
- With `OPENAI_API_KEY`: `llmQuestion()` — sends last answer + recent dialogue + JD + resume + slot theme to OpenAI, gets one natural follow-up question
- Without key: `fallbackQuestion()` — deterministic slot-aware questions with answer-length branching
- Returns `{question: string}`

## Key Architectural Decisions

- **No real SSO in demo** — the OIDC config in README is aspirational. The running app uses the custom cookie auth exclusively
- **Server Actions everywhere** — form submissions (`createAssessment`, `completeInterview`, `signOff`, `inject`, `flag`) are all Next.js Server Actions, not separate API routes
- **Transcript as JSON blob** — the full conversation + AI assessment is stored as a single JSON string in `Interview.transcriptJson`, not normalized rows
- **Observer not wired to live bot** — `ObserverEvent` records are saved but the voice client doesn't poll for them yet (MVP gap)
- **SQLite** — fine for local/demo, would need to swap to Postgres for production via `DATABASE_URL` change only (Prisma handles the rest)

## STT (Speech-to-Text) — Web Speech API issues

**Current problems:**
- Uses browser's Web Speech API (Chrome only, sends audio to Google's servers)
- No confidence threshold filtering — low-confidence words get accepted as-is
- `continuous=true` with `interimResults=true` causes Chrome to rarely fire `isFinal`, so you're stitching interim+final manually which loses words
- No silence detection — candidate has to manually click "Send answer"
- No punctuation/capitalization in transcripts, making AI context worse

**Suggestions:**

| # | Suggestion | Impact |
|---|---|---|
| 1 | **Replace Web Speech with OpenAI Whisper** — record audio via `MediaRecorder`, send the blob to a new `/api/transcribe` endpoint that calls `whisper-1`. Far more accurate, works in all browsers, handles accents, technical jargon (Kafka, Kubernetes, etc.) | High |
| 2 | **Add VAD (Voice Activity Detection)** using the `@ricky0123/vad-web` library — auto-detect when the candidate stops speaking instead of requiring a button click. Removes the biggest UX friction point | High |
| 3 | **Filter by confidence score** — Web Speech gives a confidence value per result; ignore anything below 0.6 to avoid garbage words polluting the transcript | Medium |
| 4 | **Add language hint** — pass the JD's detected domain keywords as hints to Whisper's `prompt` parameter (e.g. "Kafka, Kubernetes, PostgreSQL") — Whisper uses this to bias recognition toward technical terms | Medium |

| 5 | **Replace with OpenAI TTS** (`tts-1` model, `onyx` or `nova` voice) — stream audio via a new `/api/speak` endpoint, play with `<audio>` element. Sounds like a real interviewer, consistent across all browsers | High |

## AI Response Quality — `next-question` and `assessInterviewWithAi`

**Current problems:**
- No retry — one OpenAI failure falls back to heuristic permanently for that turn
- Transcript sent as raw text with no structure — AI has to infer speaker roles from "Interviewer:"/"Candidate:" prefixes
- `max_tokens: 200` for questions is tight — complex follow-ups get cut mid-sentence
- Assessment uses `gpt-4o-mini` for both question generation AND final scoring — scoring needs a stronger model
- No conversation memory management — sends last 14 utterances flat, no weighting of recent answers
- Temperature `0.55` for questions is fine but `0.25` for assessment can make verdicts too conservative
- The heuristic fallback scores based on character count — completely meaningless signal

**Suggestions:**

| # | Suggestion | Impact |
|---|---|---|
| 9 | **Add retry with exponential backoff** — 3 attempts before falling back to heuristic. Currently one network hiccup kills the AI for that turn | High |
| 10 | **Use `gpt-4o` for final assessment, keep `gpt-4o-mini` for questions** — assessment is a one-time call at the end, cost is negligible, accuracy difference is significant | High |
| 11 | **Increase `max_tokens` to 300-350** for question generation — prevents cut-off follow-ups on complex topics | Medium |
| 12 | **Add a `silenceHint` to the question prompt** — when the candidate's answer is very short (< 30 words), instruct the AI to ask a simpler clarifying question rather than a deep follow-up | Medium |
| 13 | **Weight recent utterances more** — instead of flat last-14, send last 4 utterances in full + a compressed summary of earlier turns | Medium |
| 14 | **Add `OPENAI_API_KEY` missing warning at interview start** — currently the user only finds out after completing the interview that scoring was heuristic | Medium (UX) |
| 15 | **Replace heuristic fallback with a rule-based scorer** — use keyword matching against JD text, answer length distribution, and question coverage (slots answered) for a more meaningful fallback score | Low-Medium |
| 16 | **Add `stream: true` to question generation** — stream the question text to the client token-by-token so TTS can start speaking the first sentence while the rest generates | Medium |
